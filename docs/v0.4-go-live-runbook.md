# v0.4 Go-Live Runbook. X402 / M2M settlement to mainnet

The single source of truth for taking the **shadow-built** x402 + escrow spine to
**first real governed machine payment on Base mainnet** (plan Phase 5). It records
what is already built, why no money can move yet, and the exact ordered path to
go live. Flagging which steps are **code** vs **human / business** decisions.

- **Plan of record:** `docs/v0.4-execution-plan.md` (phases 0–6) + RFC 0001.
- **Status (2026-05-27):** Phases 1–3 + reputation (plan Phase 6) built,
  **live-in-shadow**. Phase 4 (4337 interop) not started. Phase 5 (mainnet)
  blocked on the external audit + the steps below.

## 2. Why no money can move yet (the fail-closed layers)

Three independent gates must each be opened. **on purpose**. Before a single
mainnet payment can occur:

1. **No agent is live.** The commerce agent is **not** in `LIVE_AGENTS`
   (`services/agent-router/src/promotion-config.ts`). Both proposal entry points
   (`/agents/run` and the `/agents/events` BullMQ worker) terminate a shadowed
   financial proposal as `shadow_completed`. No `PaymentIntent` is even created.
2. **No rail is registered.** `x402_base` / `escrow_base` are not registered in
   `RailRegistry` at boot, so dispatch fails closed (`execution_rail_unavailable`).
   Stub rails additionally throw under `NODE_ENV=production`.
3. **The §6 gate checks are dormant.** Checks 3.5/5.5/6.5/6.6/8.5 record no row
   until their loaders are injected. And even then, the contract that escrow
   settlement depends on (`BrainEscrow`) is **unaudited / testnet-only**.

Every path still flows through `PaymentIntent → §6 gate → audit`. There is no
un-gated path. (Verified posture: `docs/audit/payment-runtime-verification.md`.)

## 4. Dormant → enforcing (which loader activates each gate check)

| Gate check (file: `shared/src/gate/gate.ts`) | Activated when                                                         |
| -------------------------------------------- | ---------------------------------------------------------------------- |
| 3.5 `onchain_settlement_permitted`           | policy-VM sets `onchain_settlement_permitted` on the decision          |
| 5.5 `agent_counterparty_attested`            | `attestCounterpartyAgent` injected (reads `BrainMCPAgentRegistry`)     |
| 6.5 `x402_payment_context_valid`             | intent carries `settlement` (already wired). Enforces today in shadow |
| 6.6 `escrow_state_bound`                     | `resolveEscrowState` injected (reads `BrainEscrow.getEscrow`)          |
| 8.5 `micropayment_cap_within_window`         | `sumAgentWindowSpend` injected + policy `micropayment_window_cap`      |
| reputation tightening (policy, not gate)     | `resolveReputation` injected into `PolicyService`                      |

Wiring a loader is **additive + fail-closed**: absent ⇒ dormant (canonical path
unchanged); present ⇒ the deterministic check enforces. None is an LLM/reputation
precondition (Standards §6, Principle #5).

## 5. Rollback / kill-switch

- **De-promote:** remove the agent from `LIVE_AGENTS` → instantly back to shadow.
- **Pause:** `POST /payment-intents/{id}/pause` (re-runs the live gate on resume).
- **Revoke session key / pause** on `BrainSmartAccount` (on-chain spend stops).
- **Unregister the rail** → dispatch fails closed again.
- General rollback procedure: `docs/rollback.md`.

## 7. Non-negotiables (unchanged, all phases)

Never bypass/weaken the §6 gate; never fork the payment path; no PII on-chain
(hash-only, RFC §3); no money contract to mainnet without external audit (RFC §9);
contracts immutable by default; reputation/LLM never a §6 precondition.
