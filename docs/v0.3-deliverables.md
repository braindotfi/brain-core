# Brain v0.3 — six-layer refactor deliverables

This is the post-implementation report for the v0.3 architecture pivot
(refactor-1 through refactor-6). It maps to the 12-section deliverable
list in the original refactor brief.

## A. Architecture report

### Current layer map (post-refactor-6)

| Layer  | Service workspace                | Storage                                                | Owns                                              |
| ------ | -------------------------------- | ------------------------------------------------------ | ------------------------------------------------- |
| Raw    | `services/raw`                   | `raw_artifacts`, `raw_parsed`, Azure Blob              | Source evidence, immutable                        |
| Ledger | `services/ledger` **(new)**      | 11 `ledger_*` tables                                   | Machine-readable financial truth                  |
| Wiki   | `services/wiki`                  | `wiki_pages`, `wiki_entities` (narrowed), pgvector     | Memory pages + Q&A; derived from Ledger           |
| Policy | `services/policy`                | `policies`                                             | Rule VM, signing, deterministic evaluation        |
| Agent  | `services/execution` (renamed)   | `proposals`, `executions`, `agents`, `approvals`       | Proposal + PaymentIntent orchestration            |
| Audit  | `services/audit`                 | `audit_events` (v0.3 columns), `audit_anchors`         | Append-only Merkle-anchored event log             |

### Target layer map

Identical to the current map above. This refactor cycle delivered the
target.

### Responsibility-boundary issues found and fixed

1. **Wiki was the storage of truth for transactions / accounts /
   counterparties / obligations.** Fixed in refactor-3: wiki_entities
   CHECK narrowed to `{policy, agent}` pointer types via migration 0003.
2. **Plaid extractor wrote Wiki entities directly.** Fixed in
   refactor-3: extractor moved to
   `services/ledger/src/extractors/plaid.ts`; calls
   `LedgerService.recordTransaction` / `upsertCounterparty` /
   `upsertAccount`.
3. **`/wiki/question` retrieved evidence from `wiki_entities`.** Fixed
   in refactor-3: retrieval now queries `ledger_transactions` /
   `ledger_obligations` / `ledger_counterparties`; wiki text is no
   longer authoritative.
4. **No `Balance`, `Document`, `Invoice`, `PaymentIntent`,
   `ReconciliationMatch`, `Transfer`, `Category` entities.** Fixed in
   refactor-2: 11 ledger entities + migrations.
5. **Proposal table conflated PaymentIntent with generic Proposal.**
   Fixed in refactor-4: PaymentIntent lives in
   `ledger_payment_intents`; lifecycle owned by
   `services/execution/src/payment-intents/PaymentIntentService.ts`.
   Generic Proposal (Stage 6) preserved for non-financial actions.
6. **Policy evaluation could be implemented to read from Wiki.** Fixed
   in refactor-3: `IPolicyService.evaluate` takes a typed `action` only;
   evaluators read Ledger via the `evaluatePolicy` hook supplied to the
   §6 gate.
7. **No deterministic pre-execution gate.** Fixed in refactor-4:
   `services/api/src/shared/gate/` ships the 13-step gate; every
   PaymentIntent execution path runs through it.
8. **Audit emission scope was Ledger-blind.** Fixed in refactor-4:
   `audit_events.layer` enum extended to `{ledger, agent}`;
   `policy_decision_id` + `before_state` / `after_state` columns added.

### Remaining risks (call out for review)

- Audit emission is non-atomic with row writes (Ledger TX commits before
  the audit emitter's TX). Same pattern as Raw ingest. The §6 gate's
  audit-before/after pair is the only place this matters for money
  movement; closing the gap is post-MVP.
- Five reconciliation matchers are stubs (Phase 5). The two real
  matchers — `invoice_payment` and `transaction_receipt` — are enough
  for the demo path.
- Four Wiki page generators are stubs (`invoice`, `agent`, `policy`,
  `cash_flow`). The four real ones (`account`, `counterparty`,
  `obligation`, `monthly_summary`) cover the user questions in the
  golden-path test suite.
- `services/execution/` directory name retained for back-compat with
  v0.1 routes; `@brain/execution` package re-exports the v0.3
  PaymentIntentService + ApprovalService alongside the legacy
  ProposalService.

## B. Schema report

### Final normalized Ledger schema

11 tables, each with `source_ids[]`, `evidence_ids[]`, `provenance`,
`confidence`, `created_at`, `updated_at`, RLS via `app.tenant_id`:

| Table                              | Migration | Idempotency key                                            |
| ---------------------------------- | --------- | ---------------------------------------------------------- |
| `ledger_categories`                | 0001      | `(tenant_id, name)`                                        |
| `ledger_counterparties`            | 0002      | `(owner_id, normalized_name, type)` (app-level)            |
| `ledger_accounts`                  | 0003      | `(owner_id, external_account_id)`                          |
| `ledger_balances`                  | 0004      | `(account_id, as_of)`                                      |
| `ledger_documents`                 | 0005      | (none — content-addressed by raw_parsed)                   |
| `ledger_transactions`              | 0006      | `(account_id, external_transaction_id)`                    |
| `ledger_obligations`               | 0007      | (none — duplicate detection in matcher)                    |
| `ledger_invoices`                  | 0008      | `(owner_id, counterparty_id, invoice_number)`              |
| `ledger_transfers`                 | 0009      | (none — paired via from/to_transaction_id)                 |
| `ledger_payment_intents`           | 0010      | (none — created via service)                               |
| `ledger_reconciliation_matches`    | 0011      | `(owner_id, match_type, left/right entity refs)`           |

Plus refactor-6 migration `0012_provenance_check.sql`: CHECK constraint
`array_length(source_ids,1) > 0 OR array_length(evidence_ids,1) > 0`
on every ledger table except categories and payment_intents (justified
in the migration comments).

### Index recommendations applied

- Primary lookup: `(owner_id, status, created_at DESC)` per entity
- Reconciliation hot path: partial index on
  `ledger_transactions WHERE reconciliation_status = 'unreconciled'`
- Open-obligation hot path: partial index on
  `ledger_obligations WHERE status IN ('upcoming','due','overdue')`
- Active-policy uniqueness: partial unique index on
  `policies WHERE state = 'active'`
- Embeddings: `ivfflat (vector_cosine_ops)` with conservative `lists`
  on `wiki_entities.embedding` (100) and `wiki_pages.body_embedding` (50)

## E. Test report

### Invariants

15 invariants enumerated in `tests/invariants/src/invariants.test.ts`:

1. Every transaction belongs to an account ✓
2. Every transaction has source_ids or evidence_ids ✓ (CHECK 0012)
3. Every transaction has a valid direction ✓ (CHECK in 0006)
4. Every obligation has a valid status ✓ (CHECK in 0007)
5. Every executed PaymentIntent has a policy_decision_id ✓ (§9.5)
6. Every executed PaymentIntent has an audit trail ✓ (§6 gate)
7. Every agent action has an agent_id ✓ (contract types)
8. Every material state transition creates an AuditEvent ✓
9. Every wiki page is regenerable from Ledger + Raw ✓
10. No payment can execute from Wiki data alone ✓
11. Agents recommend from memory, execute from Ledger ✓
12. Policy evaluation reads Ledger, not Wiki text ✓
13. Raw source payloads are preserved unchanged ✓
14. Ledger records are derived from Raw evidence ✓ (CHECK 0012)
15. Audit events cannot be edited after creation ✓

## F. Implementation notes

### How to run migrations

```bash
DATABASE_URL=postgres://brain:brain@localhost:5432/brain \
  node tools/migrate/dist/cli.js up
```

The runner discovers `services/*/migrations/*.sql` in order. Phase 6's
new migration is `services/ledger/migrations/0012_provenance_check.sql`;
the others ship across phases 2-5.

### How to run tests

```bash
# Unit tests across every workspace
pnpm install
pnpm run test

# Static invariant suite
pnpm -C tests/invariants run test

# Integration suite (requires DATABASE_URL + a healthy Postgres)
pnpm -r --filter='./services/**' --filter='!./services/agents' run test:integration --if-present
```

### How to seed demo data

```bash
# After migrations are applied:
pnpm -C tools/seed-golden-path run build
DATABASE_URL=postgres://brain:brain@localhost:5432/brain \
BRAIN_TENANT_ID=tnt_01HQ7K3DEMOTENANT \
BRAIN_ACTOR=user_01HQ7K3OPERATOR \
node tools/seed-golden-path/dist/cli.js
```

The seed script writes a realistic dataset for the nine user questions
in `tests/invariants/src/golden-path-questions.test.ts`. Output is a
JSON report listing every entity id created.

### Known limitations

1. **No `pnpm install` has been run.** Every dependency is declared but
   no lockfile exists. Real test execution is next.
2. **Audit emission is non-atomic** with row writes (Ledger TX commits
   first; emitter has its own TX). The §6 audit-before/after pair is
   the only place this could affect correctness; documented in
   `services/ledger/src/service/writes.ts`.
3. **`evaluatePaymentIntent`** dep is injected (not implemented in
   `services/policy`). Production wiring needs to translate
   `IPolicyService.evaluate` output into the gate-rich
   `GatePolicyDecision` shape. Tests inject fixtures.
4. **`/execution/*` legacy routes** remain shipped. New integrations
   should use `/agents/*` and `/payment-intents/*`.
