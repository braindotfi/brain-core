# Open-ecosystem interop — ERC-4337 / Coinbase Smart Wallet / CDP Paymaster (Phase 4)

Design for RFC 0001 §7.5 (the "open ecosystem path") and plan Phase 4: let an
**external** agent or user — holding an ERC-4337 / **Coinbase Smart Wallet** —
settle a Brain-mediated M2M payment **through the same `PaymentIntent` → §6 gate →
audit** path as the internal governed flow, with **gasless** UX via the **CDP
Paymaster**. This makes Brain a first-class participant in Base's account-
abstraction ecosystem without forking the control plane.

- **Status:** Draft (design). Builds shadow-first; **audit after** the full
  build, per the go-live runbook (`docs/v0.4-go-live-runbook.md`).
- **Source of truth:** RFC 0001 §7.5 (account model), §6 (gate), §3 (on-chain
  privacy), §9 (contract safety); decision **D-2** (4337 for the open surface
  only) and **D-4** (USDC on Base).

> **Non-negotiable (unchanged):** the open path **terminates in the same §6
> gate** — it is a new _source of authorization_, never a new _un-gated_ money
> path. The internal governed path keeps the custom session-key
> `BrainSmartAccount` (RFC §7.5: **no 4337 rewrite of the internal path**).

## 1. The two coexisting flows (one gate)

|                    | Internal governed path                                           | **Open ecosystem path (this phase)**                    |
| ------------------ | ---------------------------------------------------------------- | ------------------------------------------------------- |
| Who initiates      | Brain's payment agent                                            | An **external** agent / user wallet                     |
| On-chain authority | `BrainSmartAccount` **session key** (spend caps + policyVersion) | **Coinbase Smart Wallet** + **Spend Permission** (4337) |
| Gas                | paid normally                                                    | **sponsored** by the **CDP Paymaster** (gasless)        |
| Control plane      | `PaymentIntent` → §6 gate → audit                                | **identical** — `PaymentIntent` → §6 gate → audit       |
| Settlement         | x402 / escrow (USDC on Base)                                     | x402 / escrow (USDC on Base)                            |

Both converge on the gate. The gate is **source-agnostic**: it checks the
`PaymentIntent` (identity, policy, balance, the 5 x402/escrow checks), not how the
authorization was minted.

## 2. What Brain builds vs. integrates

Phase 4 is **largely off-chain orchestration**. Brain **integrates** existing,
already-deployed external infrastructure and **builds none of it**:

| External (integrate, build nothing)             | Role                                                                               |
| ----------------------------------------------- | ---------------------------------------------------------------------------------- |
| ERC-4337 **EntryPoint**                         | Standard AA mempool entry; validates/executes `UserOperation`s.                    |
| **Coinbase Smart Wallet**                       | The external user/agent's 4337 account.                                            |
| **SpendPermissionManager** (Coinbase)           | On-chain recurring-allowance grants; the spender calls `spend(permission, value)`. |
| **CDP Paymaster** (Coinbase Developer Platform) | Sponsors `UserOperation` gas → gasless.                                            |

**Brain builds (off-chain):** the _adapter_ that (a) validates an external
**Spend Permission** / `UserOperation`, (b) maps it to a `PaymentIntent`, (c) runs
the **§6 gate**, and (d) on approval, executes the settlement (the `spend` call /
a sponsored `UserOperation`) and audits it.

> **Key consequence (RFC §9):** the simple open x402 path needs **no new Brain
> money contract** — it reuses Coinbase's `SpendPermissionManager` + CDP Paymaster
> for the direct buyer→payee transfer, and the already-built `BrainEscrow` for the
> conditional case. So Phase 4 does **not** grow the contract audit surface beyond
> the x402-settlement + escrow contracts already in `AUDIT-SCOPE.md`.

## 3. The mapping: Spend Permission ↔ Brain session-key + cap

The RFC's core insight (§7.5) is that a Coinbase **Spend Permission** encodes the
_same_ authorization shape as a `BrainSmartAccount` **session key** — "spender X
may spend ≤ allowance of token T per period P, within [start, end]." The gate +
caps enforce both identically:

| Coinbase Spend Permission | `BrainSmartAccount.SessionKey`              | Meaning                    |
| ------------------------- | ------------------------------------------- | -------------------------- |
| `account`                 | (the smart account itself)                  | The funding wallet.        |
| `spender`                 | `holder`                                    | Who may pull funds.        |
| `token`                   | `allowedTargets` (the USDC contract)        | Asset (USDC, D-4).         |
| `allowance`               | `maxPerPeriod`                              | Cumulative cap per window. |
| `period`                  | `periodSeconds`                             | Rolling-window length.     |
| `start`                   | `validAfter`                                | Not before.                |
| `end`                     | `validUntil`                                | Not after.                 |
| `salt` / `extraData`      | (`policyVersion` binding is Brain-specific) | Uniqueness / policy pin.   |

Because the shapes match, the **§6 gate's existing checks already cover the open
path**: the per-window `allowance` maps onto the **micropayment cumulative cap**
(check 8.5) and the **amount-within-limit** check; the `token` onto the x402
**payment-context** check (6.5, USDC); and the external agent's identity onto
check 1 plus the **agent-counterparty attestation** (5.5, via
`BrainMCPAgentRegistry`). No new gate check is required — only that the resolver
populate the intent's settlement context + window cap from the Spend Permission.

## 4. End-to-end flow (shadow-first)

```
external agent (Coinbase Smart Wallet)
        │  signs a Spend Permission  (spender = Brain settlement addr / payee)
        ▼
Brain open-ecosystem adapter (off-chain)
        │  1. validate the permission (sig / SpendPermissionManager read):
        │     token == USDC, allowance ≥ amount, now ∈ [start,end], spender ok
        │  2. resolve → PaymentIntent { x402_settle, source = smart account,
        │     destination = payee agent counterparty, amount, currency: USDC,
        │     settlement context + window cap from the permission }
        ▼
PaymentIntent → §6 gate (identity, policy, 3.5/5.5/6.5/8.5, balance, …) → audit
        │  on approve:
        ▼
settle on Base (USDC):
   • simple:  SpendPermissionManager.spend(permission, amount) → payee
   • escrow:  spend → BrainEscrow.lock → release/refund (gate check 6.6)
        │  gas sponsored by the CDP Paymaster (UserOperation) → gasless
        ▼
execution receipt + chain_tx_hash → audit + reconciliation (onchain_settlement matcher)
```

Every step before "settle" is Brain's existing spine. The only _new_ surface is
the adapter (steps 1–2) and the gasless `UserOperation` construction.

## 5. Privacy (RFC §3) — hash-only

Only addresses, amounts, and hashes touch the chain. A Spend Permission's terms
are an on-chain Coinbase construct (addresses + amounts + period); Brain stores a
**commitment / reference**, never PII. `check-no-onchain-pii` continues to hold
(no new Brain `.sol` with `string` on the ABI).

## 6. Gasless / paymaster policy

The CDP Paymaster sponsors the settlement `UserOperation`'s gas. Brain must only
sponsor **gated** operations: the `UserOperation` is constructed **after** the §6
gate approves, so Brain never pays gas for an un-gated or rejected payment. The
CDP API integration (keys, RPC, bundler) is **deferred live-wiring** — like the
real rails' SDK construction (`X402Client` / Plaid / viem). Until configured, the
open path **fails closed**.

## 7. The §6 gate stays the single control point

- **Identity:** the external agent registers in `BrainMCPAgentRegistry` (EIP-712
  scope attestation) exactly like an internal agent → check 1 + check 5.5.
- **Authorization:** the Spend Permission's allowance/period/window flow into the
  intent and are re-checked by the deterministic gate (8.5 + amount limit).
- **No reputation/LLM precondition** (Principle #5); reputation (Phase 6) may only
  _tighten_ the policy threshold.
- **Promotion:** the open-ecosystem (commerce) agent ships **shadowed** in
  `LIVE_AGENTS` and is promoted only after `check-promotion-readiness` + the audit.

## 8. Build sequence (shadow-first; audit after)

1. **4-0 (this doc).**
2. **4-A** — pure off-chain **external-authorization model**: type a Spend
   Permission, validate it maps to a settlement (token=USDC, allowance ≥ amount,
   period covers now, spender ok), and expose the session-key-shaped projection.
   Fully unit-tested; the on-chain `SpendPermissionManager` read is injected/
   deferred.
3. **4-B** — the **open-ecosystem rail/adapter**: external authorization →
   `PaymentIntent` (x402_settle) → §6 gate → audit, settling via x402. Shadow-
   first (fails closed; agent not promoted). Proves the gate is source-agnostic.
4. **4-C** — **gasless `UserOperation` construction** via the CDP Paymaster
   (deferred SDK wiring, like the rails). Sponsorship only post-gate.

## 9. Audit posture

Per the build-everything-then-audit decision: Phase 4 adds **no new Brain money
contract** (it integrates external infra + reuses x402/escrow), so it does not
expand the contract audit scope. Any incidental on-chain helper that _did_ appear
would ship **UNAUDITED / Base Sepolia testnet-only** and be added to
`AUDIT-SCOPE.md`. Mainnet remains gated on the external audit (RFC §9).

## 10. Out of scope / assumptions

- We **integrate** Coinbase Smart Wallet / SpendPermissionManager / CDP Paymaster
  / the 4337 EntryPoint; we do not re-implement them. Their addresses + ABIs are
  configured at boot (deferred live-wiring).
- USDC on Base only (D-4). Non-USDC / fee-on-transfer tokens are out of scope.
- The internal governed path is unchanged (`BrainSmartAccount` session keys).
- D-8 (on-chain address hygiene / tenant-id rotation) is answered before mainnet,
  per the runbook — relevant here because external wallets are user-visible.

## 11. Non-negotiables

Same §6 gate (never bypassed/forked); hash-only on-chain (RFC §3); no money
contract to mainnet without external audit (RFC §9); reputation/LLM never a §6
precondition; the open path is shadow-first until promoted.
